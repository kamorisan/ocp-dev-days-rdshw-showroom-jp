= Module 1: 開発者エクスペリエンス
include::partial$_attributes.adoc[]
:imagesdir: ../assets/images

Parasol Insurance での初日です。あなたはクレーム処理チームに配属され、会社のアプリケーションプラットフォームへのアクセスを与えられました。マネージャーは、Parasol アプリケーションに新しいクレーム統計 API エンドポイントを追加するように依頼しました -- これは、ビジネスがクレームのボリュームとカテゴリを追跡するのに役立つ機能です。

何もインストールする必要はありません。チケットを提出する必要もありません。必要なものはすべて、プラットフォーム上であなたを待っています。

== 学習目標

このモジュールの終わりまでに、以下ができるようになります:

* Developer Hub をナビゲートして、アプリケーション、API、ドキュメントを見つける
* セルフサービステンプレートを使用して、分離された開発環境を作成する
* AI アシスタンスを備えたクラウドベースの Dev Spaces IDE でコードを書いてテストする
* 自動化された CI/CD パイプラインを通じてコードをプッシュし、品質ゲートの失敗を解決する
* GitOps 駆動のデプロイメントを本番環境で確認する

=== Exercise 3: クレーム統計エンドポイントを追加する

それでは機能を実装しましょう。クレーム統計 (合計、カテゴリ別およびステータス別の内訳) を返す新しい REST エンドポイントを作成します。

. Dev Spaces に戻り、左側の Explorer で `src/main/java/com/parasol/` に移動します。

. `parasol` ディレクトリを右クリックし、**New File** を選択します。名前を `ClaimsStatsResource.java` とします。
+
image::parasol-app-newfile.png[Parasol Insurance application dashboard,link=self,window=blank,width=700]

. 以下のコードを新しいファイルにコピーして貼り付けます:
+
[source,java,role="copypaste"]
----
package com.parasol;

import java.util.HashMap;
import java.util.Map;

import jakarta.inject.Inject;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.Produces;
import jakarta.ws.rs.core.MediaType;

@Path("/api/claims/stats")
@Produces(MediaType.APPLICATION_JSON)
public class ClaimsStatsResource {

    @Inject
    ClaimsResource claimsResource;

    @GET
    public Map<String, Object> getStats() {
        Map<String, Object> stats = new HashMap<>();

        System.out.println("Fetching claim statistics");

        try {
            var claims = claimsResource.getAllClaims();
            stats.put("total", claims.size());

            Map<String, Long> byCategory = new HashMap<>();
            for (var claim : claims) {
                byCategory.merge(claim.getCategory(), 1L, Long::sum);
            }
            stats.put("by_category", byCategory);

            Map<String, Long> byStatus = new HashMap<>();
            for (var claim : claims) {
                byStatus.merge(claim.getStatus(), 1L, Long::sum);
            }
            stats.put("by_status", byStatus);
        } catch (Exception e) {
        }

        return stats;
    }
}
----
+
NOTE: このコードは動作しますが、いくつかのショートカットがあります。適切なロギングフレームワークの代わりに `System.out.println` を使用し、`catch` ブロックは例外を静かに飲み込んでいます。これらのコードスメルは意図的なものです -- Section 4 で CI/CD パイプラインがそれらを検出します。

. ファイルは自動的に保存されます。Quarkus dev mode が変更を検出し、アプリケーションをリロードします -- 手動のビルドステップは不要です。

. ブラウザで、実行中の Parasol アプリケーションのタブに切り替え、URL から `dashboard.html` を削除し、ブラウザのアドレスバーの URL の末尾に `/api/claims/stats` を追加します。
+
クレームの合計と内訳を含む JSON データが表示されるはずです:
+
----
{
  "total": 15,
  "by_category": {
    "auto": 5,
    "home": 4,
    "life": 6
  },
  "by_status": {
    "open": 8,
    "closed": 7
  }
}
----
+
新しいエンドポイントが動作しています。コードを書き、ファイルを保存すると、アプリケーションは数秒でリロードされました。ビルドコマンドも、再パッケージングも、再デプロイも不要です。
+
image::parasol-claims-stats-devmode.png[Claims statistics endpoint returning JSON data in dev mode,link=self,window=blank,width=700]

=== Exercise 3: パイプラインの失敗を診断する

パイプラインが何かを検出しました。以前に行ったコードのショートカットのために、SonarQube 品質ゲートが失敗しました。

. 失敗した `sonar-scan` ステップをクリックして詳細を確認します。

. パイプラインログ出力で **SonarQube report URL** を見つけます。
+
image::ocp-pipeline-sonar-fail.png[SonarQube report showing code smell violations,link=self,window=blank,width=700]

. SonarQube レポートを開いて、具体的な問題を確認します:
* **`System.out.println` の使用** -- 本番コードでのロギングには標準出力を使用すべきではありません
* **空の catch ブロック** -- 例外を静かに飲み込むべきではありません
* **Field injection** -- SonarQube はテスト容易性を向上させるために、`@Inject` field injection よりも constructor injection を推奨しています
+
これらは提案ではありません。厳格な品質ゲートです。これらの違反があるコードは先に進むことができません。
+
image::ocp-pipeline-sonar-report.png[SonarQube report showing code smell violations,link=self,window=blank,width=700]

import java.util.Map;

import io.quarkus.logging.Log;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.Produces;
import jakarta.ws.rs.core.MediaType;

@Path("/api/claims/stats")
@Produces(MediaType.APPLICATION_JSON)
public class ClaimsStatsResource {

    private final ClaimsResource claimsResource;

    public ClaimsStatsResource(ClaimsResource claimsResource) {
        this.claimsResource = claimsResource;

    @GET
    public Map<String, Object> getStats() {
        Map<String, Object> stats = new HashMap<>();

        Log.info("Fetching claim statistics");

        try {
            var claims = claimsResource.getAllClaims();
            stats.put("total", claims.size());

            Map<String, Long> byCategory = new HashMap<>();
            for (var claim : claims) {

            stats.put("by_category", byCategory);

            Map<String, Long> byStatus = new HashMap<>();
            for (var claim : claims) {

            }
            stats.put("by_status", byStatus);
        } catch (Exception e) {
            Log.error("Error fetching claim statistics", e);
        }

        return stats;
    }
}

== モジュールのまとめ

Parasol Insurance での完全な開発者ジャーニーを完了しました -- オンボーディングから本番デプロイメントまで -- チケットを提出したり、ツールをインストールしたり、誰かを待ったりすることなく。

**達成したこと:**

* Developer Hub のソフトウェアカタログを通じてアプリケーション、API、ドキュメントを発見した
* Lightspeed を使用してプラットフォームに関する AI を活用した回答を得た
* セルフサービステンプレートで分離された開発環境を作成した
* Dev Spaces で事前設定されたクラウド IDE でコードを書いてテストした
* AI コーディングアシスタントを使用してコード品質の問題を修正した
* 品質ゲートを備えた自動化された CI/CD パイプラインを通じてコードを push した
* リリースタグを作成し、本番環境にデプロイするための GitOps 昇格を承認した
* Developer Hub のトポロジビューを通じて本番環境で機能を確認した

**重要なポイント:**

* **Developer Hub** は単一のガラス窓です -- カタログ、テンプレート、CI/CD の可視性、AI 支援が 1 か所に
* **Dev Spaces** はローカル環境のセットアップを排除します -- すべての開発者が同じ事前設定されたワークスペースを取得
* **Pipelines** は品質を自動的に強制します -- コードスメルとセキュリティ問題は本番環境の前に検出される
* **GitOps** はデプロイメントを再現可能で監査可能にします -- Git が唯一の信頼できる情報源
* **プラットフォーム** は差別化されていない重労働を処理するため、開発者はコードの記述に集中できる

これが、統合されたアプリケーションプラットフォームの姿です。コードから本番環境まで、必要なものすべてが、オンプレミス、あらゆるクラウド、またはエッジで利用可能です。
